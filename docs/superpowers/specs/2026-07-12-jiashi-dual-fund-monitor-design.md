# 嘉实双基金组合监控 Skill 设计

日期：2026-07-12  
状态：待用户书面复核  
目标仓库：`hawtim/skills`  
目标目录：`A-share/jiashi-dual-fund-monitor/`

## 1. 目标与边界

建立一个可复用的嘉实双基金组合监控 skill，并由每日、每周、季度三个独立定时任务调用。系统负责读取交易记录、获取公开市场与基金数据、计算组合状态、检查预设规则、生成 Markdown 报告并同步到 GitHub。

系统只做数据采集、计算、状态判断、建议生成、历史记录和复盘。它不自动申购、赎回或执行其他外部交易；所有交易动作必须由用户人工确认，实际成交后才写入交易 CSV。

### 1.1 公开信息边界

skill、交易 CSV、报告、自动化提示词和 GitHub 提交只描述基金组合、计划额度、交易记录、市场数据、组合表现和监控判断。个人财务信息不属于该系统的数据模型或输出范围。原始导出文档仅作为本次设计输入，不复制到 skill 或 GitHub。

## 2. 固定标的与组合规则

| 基金代码 | 固定名称 | 组合角色 | 初始计划 | 最终上限 |
|---:|---|---|---:|---:|
| 000043 | 嘉实美国成长股票（QDII） | 美国大盘成长核心仓 | 150,000 元 | 200,000 元 |
| 017730 | 嘉实全球产业升级股票发起式（QDII）A | 全球科技与半导体增强仓 | 50,000 元 | 100,000 元 |
| 合计 |  | 科技成长组合 | 200,000 元 | 300,000 元 |

`017731 C` 不作为可选持仓，不出现在目标配置、交易模板或报告状态表中。`017730 A` 当前购买渠道显示 0.00% 申购费，但该优惠不得写成永久产品规则；实际交易费用以每笔成交确认记录为准，准备新增交易时应重新核对渠道费率。

后续 100,000 元计划额度默认分两档：

| 触发阶段 | 000043 | 017730 A | 合计 |
|---|---:|---:|---:|
| 第一档：相对成本回撤 8%—10%，且其他检查通过 | 25,000 元 | 25,000 元 | 50,000 元 |
| 第二档：相对成本回撤约 15%，且投资逻辑未被破坏 | 25,000 元 | 25,000 元 | 50,000 元 |

相对成本回撤 5%—8% 只进入观察，不产生自动加仓结论。若一个月内没有明显回撤，但趋势、策略和基本面正常，报告可以提出“时间型建仓复核”，但仍需人工确认且不得突破单只基金或组合上限。

## 4. 交易 CSV 与组合计算

`data/operation-log.csv` 是唯一的交易事实来源，初始只有表头，不虚构初始交易。建议字段：

```text
record_id,reported_at,trade_date,confirmation_date,account_label,action_type,trigger_type,fund_code,fund_name,share_class,side,confirmed_amount,confirmed_shares,confirmed_nav,fee,currency,total_invested_after_trade,position_cost_after_trade,position_shares_after_trade,notes,source
```

规则：

1. 只记录用户提供或确认的实际成交。
2. 申购以确认金额、确认份额、确认净值和实际费用为准；不能用下单日估值替代确认数据。
3. 支持申购、赎回、分红再投资和现金分红；未发生的动作不预填。
4. 后续用户提供不同 App 或平台导出的 CSV 时，先映射到规范字段并校验，再追加到 canonical log；保留原始来源字段，不覆盖原始文件。
5. 同一成交通过 `record_id` 及代码、日期、金额、份额组合检查重复。
6. 报告使用最新可得净值重建份额、累计投入、加权成本、当前市值、未实现收益、已实现收益、组合权重、距目标上限差额和阶段回撤。
7. CSV 和报告只使用本节定义的投资记录字段，不扩展个人信息字段。

计算脚本只接受本地 CSV 和显式传入的净值/基准快照，不把联网抓取与组合计算耦合在一起。这样可对计算逻辑做确定性测试，并允许自动化在不同数据源之间降级。

## 5. 监控规则

每日判断按以下顺序执行：

1. **数据完整性**：确认交易日志、最新基金净值、数据日期和基准日期；QDII 净值存在时差时必须明确标注。
2. **价格条件**：检查各基金相对成本回撤、组合距阶段高点回撤及是否接近 5%—8%、8%—10%、约 15%触发区。
3. **基金条件**：检查基金经理、公告、限购、规模异常、策略定位、相对基准表现和最新披露。
4. **基本面条件**：检查美国大型成长盈利预期、AI 资本开支、半导体周期以及影响成长估值的利率和汇率环境。
5. **组合条件**：检查 20:10 目标结构、两只基金持仓重合度、增强仓是否接近或超过组合约三分之一、单只及组合计划上限。
6. **动作分类**：只输出 `无动作`、`观察`、`加仓候选`、`暂停加仓`、`再平衡复核` 或 `减仓/替换复核`。

加仓候选必须同时满足价格、基金策略、基本面和组合风险检查。单日或单周下跌、新闻标题、一次跑输基准或普通 5%—10%波动不能单独触发减仓。

以下事件升级为重点复核：

- 基金经理变更或管理框架明显改变；
- 000043 显著偏离美国大盘成长/罗素 1000 成长增强定位；
- 017730 A 失去跨市场、跨子行业轮动特征并变成单一方向高度集中；
- 连续两个完整披露季度明显跑输各自基准；
- 两只基金前十大持仓或风险暴露重合度显著上升；
- 增强仓因上涨接近组合一半；
- 产品评分低于 65 时暂停激进新增，低于 50 时进入减仓或替换复核。

## 6. 数据源与时效

优先使用公开、可核验且日期明确的来源：

1. 嘉实基金官网及正式产品公告、产品资料概要、招募说明书、季度/半年度/年度报告；
2. 基金净值公开页面或可靠基金数据源；
3. IWF 作为 000043 的可交易基准代理，同时说明它不是基金法定基准本身；
4. SOXX、SMH、纳斯达克及可得全球科技指数作为 017730 A 的产业代理，不把单一代理当作完整法定基准；
5. 美债收益率、美元兑人民币、美国大型科技与半导体公司公开披露及权威宏观来源。

所有报告写明 `数据截至时间`。每日 08:30 报告应区分“最新已公布基金净值”和“隔夜市场代理表现”。若净值、交易记录或关键公告不可用，生成数据不足报告并降级为观察，不给具体交易金额。

## 7. 三类报告与定时任务

### 7.1 每日报告

- 时间：每个工作日 08:30，Asia/Shanghai。
- 文件：`reports/daily/jiashi-dual-fund-daily-YYYY-MM-DD.md`。
- 内容：一句话结论、数据时点、组合状态、两只基金状态、基准与隔夜市场、重大公告/事件、规则检查、今日动作、下一触发点和数据缺口。
- 周末和明确非交易日不生成正常日报；无法确认节假日时可生成简短状态报告并标注。

### 7.2 每周报告

- 时间：每周六 10:00，Asia/Shanghai。
- 文件：`reports/weekly/jiashi-dual-fund-weekly-YYYY-MM-DD.md`。
- 内容：周度收益与回撤、相对基准、两只基金相对强弱、组合权重偏离、触发位、宏观与产业变化、下周观察重点及是否需要人工深度复盘。

### 7.3 季度报告

- 时间：在基金法定披露窗口内周期性检查；发现尚未处理的新报告期后只生成一次。
- 文件：`reports/quarterly/jiashi-dual-fund-quarterly-YYYY-QN.md`，半年度或年度披露在标题中注明报告类型。
- 内容：前十大持仓变更、行业与国家配置、股票仓位、规模、换手率、经理观点、相对基准、两基金重合度、风格漂移、产品评分和下一季度风险预算。
- 通过自动化 memory 和目标报告文件双重去重；未发现新披露时记录跳过，不生成空季度报告。

三个自动化相互独立，分别维护运行记忆、最后处理的数据日期、报告路径、GitHub 同步结果和失败原因。任何一个失败不阻塞其他两个任务。

## 8. GitHub 同步与失败处理

每次成功生成报告后：

1. 只处理本次报告以及本次确有必要更新的嘉实 skill 文件或交易日志。
2. 不纳入仓库内其他未提交或未跟踪变更。
3. 本地报告始终先安全落盘。
4. 按现有 A 股科技 ETF 自动化约定，GitHub 写入仅使用 GitHub Connect，不以 SSH/HTTPS `git push` 作为自动化回退。
5. 单文件报告优先使用 contents API；多文件需要保持一致时使用原子 blob/tree/commit/ref 写入。
6. 输出和 automation memory 记录远端 commit SHA；连接器失败时记录原因，不把本地生成成功描述成已同步。
7. 同名报告已存在时不覆盖：使用实际运行时间或 `-rerunN` 后缀，并在报告顶部说明原计划时点与补跑原因。

## 9. 测试与验收

计算脚本采用测试先行，至少覆盖：

- 空交易日志；
- 单笔申购后的份额、成本和剩余额度；
- 多次加仓后的加权成本；
- 赎回、费用和已实现收益；
- 重复成交拒绝；
- `017731` 或未知代码拒绝进入目标组合；
- 单只与组合上限检查；
- 5%—8%观察、8%—10%第一档、约 15%第二档；
- 价格满足但策略/基本面失败时不得给加仓候选；
- QDII 净值滞后和关键数据缺失时降级；
- 公开文件只能出现已批准的基金、交易、市场与组合字段；任何额外个人字段都会使测试失败。

skill 验证包括：

- `SKILL.md` 与 `agents/openai.yaml` 格式校验；
- 三种典型调用的基线测试和启用 skill 后测试；
- 三类报告模板字段完整性；
- 自动化提示词路径、时区、去重、GitHub 同步和失败降级检查；
- 在不修改交易日志的情况下完成一次空仓模拟运行。

## 10. 完成标准

实现完成需同时满足：

1. 一个可发现、可验证的 `jiashi-dual-fund-monitor` skill 已加入仓库；
2. 固定标的是 `000043` 与 `017730 A`，不存在 `017731` 目标配置；
3. canonical 交易 CSV 初始为空且支持未来 App CSV 映射；
4. 计算与规则测试全部通过；
5. 每日、每周、季度三个自动化按确认时间启用；
6. 三类报告可写入各自目录并同步到 `hawtim/skills`；
7. 所有公开产物通过字段白名单与隐私检查，只包含基金组合监控所需信息；
8. 自动化不执行交易，所有动作均要求人工确认。
